# ADR: 仕様と設計の分離および開発フローの再定義

## ステータス
承諾済み

## コンテキスト
`AGENTS.md` において、「Specs First」の原則と実際のディレクトリ構成、および開発フローの整合性が取れていなかった。特に、API設計やDB設計が「仕様（真実）」なのか「設計（実現方法）」なのかの境界が曖昧であり、SSOT（Single Source of Truth）の原則が揺らいでいた。

## 意思決定
システムの「振る舞い」と「実現構造」を明確に分離するため、以下の構成およびフローを採用する。

1. **役割の分離**:
   - **`specs/` (仕様/What)**: システムがどう振る舞うべきかを定義する。ビジネスルール、ユースケース、論理仕様を格納する。
   - **`docs/` (設計/How)**: 仕様を技術的にどう実現するかを定義する。API設計、DB設計、UI設計、インフラ詳細設計を格納する。

2. **開発フローの直線化**:
   - **Phase 1: 仕様定義**: `specs/` 配下で要件と論理を確定させる。
   - **Phase 2: 詳細設計**: `specs/` を受け、`docs/interaction`（相互作用設計）を起点に API/DB/UI/Infra の詳細を詰める。
   - **Phase 3: 実装**: 設計に基づきコード生成（Orval/Prisma）と TDD を行い、仕様への適合性を検証する。

3. **インフラ設計の明文化**:
   - `docs/infra` を新設する。相互作用設計から派生する具体的なインフラ仕様（S3のストレージクラス、ネットワーク構成、リソース制限等）を、コード（Terraform）に落とし込む前の設計図として管理する。

## 影響

### メリット
- **責務の明確化**: どのファイルに何を書くべきかが一意に定まり、AIによる開発支援の精度が向上する。
- **トレーサビリティの向上**: 仕様（What）がどのように設計（How）に変換されたかの経緯が、ID（uc-* -> api-* 等）を通じて追跡可能になる。
- **SSOTの維持**: 技術的な変更（APIのフィールド追加など）が、ビジネスロジック（仕様）を汚染することを防げる。

### デメリット / リスク
- **管理コストの増加**: 仕様と設計の両方をメンテナンスする必要がある。
- **同期のオーバーヘッド**: 小規模な変更であっても、Specs -> Docs -> Code の順序を守る厳格さが求められる。

## 備考
本決定に基づき、`AGENTS.md` のディレクトリ構成図および Mermaid によるフロー図を更新済みである。
今後、画像アップロード機能などの具体的な機能開発はこのフローを遵守して進める。
